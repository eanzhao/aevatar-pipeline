Aevatar Runtime

Aevatar Pipeline

这页讲 Aevatar 的共用执行骨架：请求怎么进入 actor system，怎么先确定定义，再交给 workflow runtime 或 scripting runtime 执行，最后再被 CQRS 投影成可消费输出。

Actor Model GAgent Runtime

命令、状态和执行生命周期都围绕 actor 展开。

Definition / Runtime Workflow / Script

先确定 definition，再创建 runtime。两套能力都遵守这个模式。

CQRS Projection / AGUI

write side 在 actor 内，read side 由投影生成。

模型 三层结构 主链路 组件 边界

Model

核心结构

Actor model 是执行根

请求不会直接打进一个万能对象。Host 先把输入规范化成命令，Application 再把命令路由到对应的 actor。

Definition / Runtime 是两类事实源

`WorkflowGAgent` / `ScriptDefinitionGAgent` 保存“跑什么”。 `WorkflowRunGAgent` / `ScriptRuntimeGAgent` 保存“这次怎么跑”。

CQRS 把读写分开

actor 持有 write side，Projection 生成 read model； workflow 当前的实时前端事件，再由 AGUI 适配层继续往外推。

Scripting 不是旁支

script 也走 definition actor、catalog、runtime actor、projection 这条主线。只是 workflow 跑 step，script 跑脚本包。

Lanes

三层结构

Control Plane

先决定这次到底跑什么

Host 和 Application 先处理入口、命令归一化、definition 解析和 run target 建立。 这一层决定“这次 run 到底要跑哪份定义”。

Execution Plane

真正执行都发生在 runtime actor 内

`WorkflowRunGAgent` + `WorkflowExecutionKernel` 跑 workflow， `ScriptRuntimeGAgent` 跑脚本。变量、超时、重试和挂起状态都落在 actor-owned state。

Read Plane

输出不是顺手拼的，而是投影出来的

run actor 发出的 runtime envelopes 会先被 Projection 消费，整理成 read model； 对当前 workflow capability 来说，前端实时事件再由 AGUIAdapter 往 SSE / WebSocket 链路继续映射。

Flow

Aevatar 如何处理一次请求

自然语言请求 判定目标 拿到定义 创建运行 actor workflow / script 执行 读侧输出

这条链路讲的是：Aevatar 怎么把一句自然语言请求接住，再在关键节点分成 workflow 和 scripting 两条执行分支。当前仓库里的 `/api/chat`、`resume`、`signal` 这组 capability API 直接对应 workflow 分支；但从 Aevatar 的整体架构看，scripting 也走同样的 definition/runtime 模式。

01 自然语言请求先到 Host

这里说的“请求”，大多数时候其实就是一句自然语言，例如“帮我分析这份需求”或者“给我生成一个 workflow”。这类请求通常通过 `POST /api/chat` 进入 Aevatar；`GET /api/ws/chat` 只是换成 WebSocket 交互，`POST /api/workflows/resume` 和 `POST /api/workflows/signal` 则是给一条已经存在的 run 发控制命令。

技术上，`Aevatar.Workflow.Host.Api` 会先做字段校验、默认值补齐和协议适配，把外部输入整理成内部命令模型，例如 `WorkflowChatRunRequest`。这一层只负责接入和协议输出，不负责保存 workflow、script 或 run 的业务状态。

02 Application 先判断这句请求该怎么接

进入框架内部之后，这句自然语言不会立刻触发 step 执行。Application 会先判断：这是一条新请求，需要新开一次 run；还是一条控制请求，要恢复一个挂起中的 run，或者给运行中的 actor 发 signal。

技术上，这一步会进入 command pipeline，由 resolver 和 binder 决定目标 actor、projection lease 和 live sink 应该怎么挂。对当前这组 capability API 来说，chat 请求会先被折叠成 workflow run target，`resume` 和 `signal` 则直接指向已有 run actor。它解决的是“由谁接手这次请求”，不是“马上调用哪个模型”。后面所有状态变化都会围绕 actor id 展开。

03 先把这次运行要用的 definition 绑定准确

这一步要解决的不是“有没有流程可跑”，而是“这次到底该跑哪一个，而且要跑对”。如果这次请求走 workflow，系统会先把正确的 YAML 和 definition 绑定定位出来。来源可以是 registry workflow、inline YAML，或者已经绑定 definition 的 source actor；如果用户显式指定了 workflow name，还要确认它和 source binding 是一致的，避免跑错流程。

技术上，named workflow 往往会被解析到稳定的 definition actor，例如 `workflow-definition:{workflow_name_lower}`。如果这次请求走 scripting，`script revision` 指的是同一个 `scriptId` 下某一版脚本定义的版本号，不是另一条执行链路。更准确的说法是：先确定这次 run 要用哪个 `definitionActorId`，必要时再带上 `requestedRevision` 作为版本约束；catalog 维护的是“当前激活哪个 revision”，runtime 真正执行时还是拿 `definitionActorId + requestedRevision` 去 `ScriptDefinitionGAgent` 查询 definition snapshot。

04 再为这次执行准备一个 runtime actor

拿到 definition 之后，Aevatar 不会直接在 definition actor 上执行，而是会把真正的执行放到 runtime actor 里。这样做的好处很直接：同一份 definition 可以反复复用，而每次运行的暂停、恢复、失败和重试都能彼此隔离。

workflow 这边是一 run 一个 `WorkflowRunGAgent`；scripting 这边则会准备一个 `ScriptRuntimeGAgent` 来承接这次脚本运行，默认也会生成新的 runtime actor id。`RunId`、变量、挂起点、重试、超时、最终结果和错误信息，都会落在 runtime actor 里。技术上，workflow command path 还会在真正派发开始命令之前，把 projection lease 和 live sink 先挂好。

05 runtime actor 开始真正执行

到这一步，runtime actor 才开始真正干活。workflow 路径里，`WorkflowExecutionKernel` 会从 `StartWorkflowEvent` 开始推进主循环，把流程一步步往前推。每到一个 step，就会发出 `StepRequestEvent`；对应模块处理完之后，再回一个 `StepCompletedEvent`，内核据此决定下一步该走哪里。

具体执行 step 的，是一组按类型分工的 modules。`connector_call` 由 connector 模块去调用外部能力；`wait_signal`、`human_input`、`human_approval` 会把 run 挂起，等后续 `signal` 或 `resume` 再继续；`llm_call` 如果配了 `target_role`，通常会交给对应的 run-scoped `RoleGAgent`，如果配了 `agent_type`，也可以解析到自定义 `IAgent`。也就是说，workflow 不是“把 YAML 从上到下读一遍”，而是一个事件驱动的调度循环。

scripting 路径里，`ScriptRuntimeGAgent` 会先查询 definition snapshot，拿到脚本源码和 revision 约束，再在 runtime 里编译并执行脚本逻辑。换句话说，第 5 步不是简单说一句“这里分岔”就结束了，而是 workflow runtime 和 script runtime 各自进入自己的执行引擎。

06 执行过程中的状态和事件持续写回 actor

运行不是黑箱，也不是“先跑完、最后再统一写一次结果”。每一步开始、完成、失败、挂起或恢复，runtime actor 都会把状态变化持续写回自己的事实层。这样做的意义很直接：一条 run 就算中途停住，系统后面也知道它停在哪、还在等什么、为什么失败，恢复时也有地方可接。

对 workflow 来说，这一层就是 run actor 自己的 write side。开始执行时会先落 `WorkflowRunExecutionStartedEvent`，结束时会落 `WorkflowCompletedEvent`；而各个模块自己的运行态，会通过 `WorkflowRunState.ExecutionStates` 继续挂在 run actor 上。像 `wait_signal`、`human_input`、`human_approval` 这种会让流程停住的节点，真正让 run 可恢复的，不是 controller 记住了什么，而是这些 pending 状态已经写回 actor 事实层。

scripting 也是同一个思路，只是写回的东西不完全一样。`ScriptRuntimeGAgent` 先把待完成的 definition query 记成 `ScriptDefinitionQueryQueuedEvent`，等 definition snapshot 到手后，再把脚本执行产出的 `ScriptRunDomainEventCommitted` 一批批写回去；这些 committed event 里会带上 `definitionActorId`、`revision`，以及最新的 state/read-model payload。要注意，这里说的是写回 actor 自己的事实层，不是说每一条 SSE 帧或前端事件都会原样持久化；前端实时流和 projection 是后面第 7、8 步的事。

07 Projection 把 write side 翻成可查询的读侧数据

Projection 不负责推进业务，它负责“翻译”。它消费 runtime actor 发出的 runtime `EventEnvelope` stream，把原本偏执行侧的消息整理成更适合查询和展示的读侧结果。对 workflow 来说，最直接的产物就是 snapshot、timeline 和 graph 这些查询模型。

技术上，workflow projection 当前是 run-isolated 语义，projection root 通常就是 runtime actor id。`WorkflowExecutionReadModelProjector` 会把同一条 envelope stream 投到 `WorkflowExecutionReport`，再由 query service 映射成 `/api/actors/{actorId}`、`/timeline`、`/graph-*` 这些接口返回的数据。你在页面上看到的执行时间线，不是 controller 临时拼出来的，而是 projection reducer 一步步累出来的读模型。

还要再分清一层：workflow 这里不只有“查询模型投影”这一条支路。同一条 run actor envelope stream 还会被 `WorkflowExecutionRunEventProjector` 映射成 `WorkflowRunEventEnvelope`，挂到 live sink / session event hub 上，后面第 8 步的 SSE 和 WebSocket 才能继续往外推。也就是说，Step 7 的重点不是“把 Event Store 再查一遍”，而是用同一条 runtime stream 同时产出查询侧和流式侧两类读结果。

08 界面和外部系统最后消费这些结果

到这一步，前端和外部系统才真正看到“结果”。有些结果是实时流出来的，例如文本增量、step 开始/结束、等待 signal；有些结果是查询出来的，例如当前 snapshot、timeline、graph，或者 run 完成后的最终状态。

技术上，这里说的前端实时输出链路，当前主要是 workflow capability。SSE 和 WebSocket 往外推送的，是一串稳定的 `WorkflowRunEventEnvelope`，不是 actor 内部状态本身，也不是运行时原始 `EventEnvelope`。前端通常只按事件类型消费它们，比如文本增量、step 开始/结束、工具调用、人工输入请求，或者最后的状态快照。

再具体一点：SSE 路径在 accepted 之后会先写一条 `aevatar.run.context` frame，后面持续跟真正的运行事件；收尾阶段还会由 finalize emitter 补一条 `state_snapshot`。WebSocket 路径则先回 command ack，再持续发送同样这套运行事件 envelope。如果外部系统不想订阅流，也可以直接走 `/api/actors/*` 这组 read-side 查询接口拿 snapshot、timeline 或 graph。scripting 也有 projection/read model，但当前仓库里没有一条和 workflow 完全同构的 AGUI 前端事件链路。

Components

关键组件

WorkflowGAgent

workflow 的 definition actor。绑定并校验 YAML，但不直接跑步骤。

WorkflowRunGAgent

workflow 的 run actor。保存 `RunId`、`Status`、`FinalOutput`、`ExecutionStates` 等运行事实。

WorkflowExecutionKernel

workflow 的主循环，负责 `StartWorkflowEvent -> StepRequestEvent -> StepCompletedEvent -> WorkflowCompletedEvent`。

RoleGAgent

按 YAML `roles` 派生的子 actor，用来承接 `llm_call` 等角色化任务。

WorkflowStepTargetAgentResolver

把 `target_role` 或 `agent_type` 解析成真实目标 actor，可挂接自定义 `IAgent`。

Projection + AGUI

`Projection` 生成 read model；workflow 当前的 `AGUI` 适配层，则把同一条运行时消息流继续映射成前端可消费的事件 envelope。

Scripting Lane

脚本能力的关键 actor

ScriptDefinitionGAgent

脚本定义事实源。负责 upsert 脚本源码、编译脚本包，并记录 read-model schema 的声明和校验状态。

ScriptCatalogGAgent

脚本版本目录。负责维护每个 `scriptId` 当前激活的 revision，也就是当前生效的脚本版本，以及 promote / rollback 结果。

ScriptRuntimeGAgent

脚本运行事实源。收到 run 请求后，会先查询 definition snapshot，再执行真正的脚本运行。

ScriptEvolutionManagerGAgent

记录脚本演进提案的生命周期，包括 proposed、validated、promoted 和 rolled back。

Boundaries

责任边界

层 负责什么 不负责什么
Host / Protocol HTTP、WebSocket 入口；SSE / 查询输出；请求规范化；协议输出。 不编排 workflow / script，不维护 actor 事实状态。
Application 解析 source、决定 run target、建立 projection lease、挂接 live sink。 不直接执行 step，不直接持有 workflow / script 的运行事实。
Core definition actor、run actor、执行内核、脚本 runtime、内置 modules。 不做协议层适配，不提供查询接口。
Infrastructure workflow 文件加载、actor port runtime adapter、capability endpoint 装配、报告落盘。 不维护 workflow 事实边界，不替代 Application 编排 run。
Projection run-isolated read model、timeline、graph、snapshot。 不直接决定业务推进。
AGUI Adapter 消费 run actor envelope，并映射成 `WorkflowRunEventEnvelope`。 不直接读 Event Store，也不直接读 actor state 拼 UI。

Pipeline

一次请求如何穿过 Aevatar

别把 Aevatar 理解成“收到请求就直接跑代码”。它会先收请求，判断这次到底要跑 workflow、 script，还是只是给现有 run 发控制命令；然后再开独立的 runtime actor，把结果投影成查询数据和实时事件。

Flow

主执行链路

这条链路先讲共同骨架，再在中间点出 workflow 和 scripting 两条执行分支。看完之后，基本就能知道 Aevatar 是怎么把一句自然语言请求接住、跑完、再把结果送出来的。

01 自然语言请求先进入 Host

这里说的请求，通常是一句自然语言，比如“帮我总结这段内容”或者“给我生成一个 workflow”。它会先通过 `POST /api/chat` 之类的入口进入 Host；`resume` 和 `signal` 则是控制已有 run 的技术接口。

Host 会先做参数校验和协议适配，把外部输入整理成内部命令模型。这一层只负责接入和输出格式，不保存 workflow、script 或 run 的业务状态。

02 Application 先判断这句请求该怎么接

Aevatar 的执行根是 actor。Application 会先判断这是要新开一次运行，还是去恢复、控制一条已经存在的 run。

技术上，这一步会把请求送进 command pipeline，由 resolver / binder 决定目标 actor 和观察链路怎么挂。它先解决“谁来接这单”，而不是“马上怎么执行”。

03 先找到对的 workflow，或者对的 script revision

workflow 分支会先确认这次到底该跑哪一个 workflow，而且要跑对。来源可能是 registry、inline YAML，或者某个已经绑定了 definition 的 source actor。

scripting 分支则先从 `ScriptCatalogGAgent` 查当前激活 revision，再由 `ScriptDefinitionGAgent` 提供源码、脚本包和 schema。这一步本质上是在确定 definition。

04 再为这次运行单独开一个 runtime actor

拿到 definition 之后，系统不会直接在 definition actor 上开跑。workflow 会新建 `WorkflowRunGAgent`；scripting 会用 `ScriptRuntimeGAgent`。

这样每次 run 都有自己的状态边界。变量、挂起点、重试、超时和最终结果，都会放在这个 runtime actor 里。

05 runtime actor 开始真正执行

workflow 路径里，`WorkflowExecutionKernel` 会推进 step 主循环：发 `StepRequestEvent`，等模块回 `StepCompletedEvent`，再决定下一步。`llm_call`、`connector_call`、`wait_signal`、`human_*` 都是在这里真正发生的。

scripting 路径里，`ScriptRuntimeGAgent` 会先查询 definition snapshot，再执行脚本逻辑。也就是说，这一步不是笼统地“分岔”，而是两个 runtime 分别进入自己的执行引擎。

06 执行过程中的状态和事件持续写回 actor

每次步骤开始、完成、失败、挂起或恢复，runtime actor 都会把状态变化和运行事件写回自己的事实源。

所以 write side 在 actor 里，不在 controller、UI 或临时 service 里；恢复执行时也靠这些已经写回去的事实继续推进。

07 Projection 把 write side 翻成可查数据

Projection 消费的是 run actor 发出的 runtime `EventEnvelope` stream，再整理成 timeline、graph、snapshot 这类读侧模型。

这就是 Aevatar 里的 CQRS 分界线：actor 负责写侧，projection 负责读侧。页面上看到的执行过程，是投影后的结果。

08 界面和外部系统最后消费这些结果

最后，AGUIAdapter 会把 envelope 映射成前端能直接消费的运行事件，SSE 和 WebSocket 负责实时往外推。

如果要查当前 snapshot、timeline 或 graph，则走 `/api/actors/*` 这组读侧接口。用户最终看到的是这些映射和投影后的输出，而不是 actor 内部状态本身。

Components

关键组件

WorkflowGAgent

workflow 的 definition actor。只管定义，不负责这次 run 真正怎么跑。

WorkflowRunGAgent

workflow 的 run actor。一次 run 一个 actor，持有这次执行的全部事实。

WorkflowExecutionKernel

workflow 的主循环。负责 step dispatch、retry、timeout 和最后收敛。

RoleGAgent

按 roles 派生的子 actor，专门承接 `llm_call` 这类角色化任务。

ScriptDefinitionGAgent

脚本定义事实源，负责保存脚本源码、revision 和 read-model schema 信息。

ScriptRuntimeGAgent

脚本运行 actor。先拿 definition snapshot，再执行脚本逻辑。

ScriptCatalogGAgent

维护脚本当前激活 revision，负责 promote 和 rollback。

ScriptEvolutionManagerGAgent

跟踪脚本演进提案，从 build、validate 到 promote / rollback。

Projection / AGUI

围绕 run actor id 生成 snapshot、timeline、graph 和流式事件。
